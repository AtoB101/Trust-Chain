==================================================
1. 项目定位（最终统一口径）
==================================================
1) 定位：
- TrustChain = Agent-to-Agent 非托管可验证支付结算协议（Protocol）

2) 非定位：
- 不是钱包（Wallet）
- 不是交易所（Exchange）
- 不是中心化托管平台

3) 核心价值：
- 资金主权在用户钱包
- 账单状态驱动资金锁定与释放
- 可验证、可追溯、可撤销、可仲裁

==================================================
2. 业务目标与设计原则
==================================================
业务目标：
1) 用户可一键进入锁仓模式（资金不离开钱包）
2) 锁仓后生成 1:1 映射账单额度（无转移价值，仅状态凭证）
3) 仅账单流程可触发结算划转
4) 非账单划转请求一律无效并上链记录为攻击意图
5) 高并发下保持资金与映射状态强一致

设计原则：
1) Non-custodial first（非托管优先）
2) State machine first（状态机优先）
3) Invariant first（不变式优先）
4) Auditability first（可审计优先）
5) Fail-safe default（失败默认安全）

==================================================
3. 术语定义
==================================================
1) 真实资金（Real Funds）
- 用户钱包中的真实代币（如 USDT/USDC）

2) 锁仓状态（Lock Mode）
- 用户授权协议在账单规则下锁定部分资金状态，不转移资产托管权

3) 映射账单币（Mapped Bill Unit）
- 1:1 映射的账单状态额度，不可转移、无市场价值，仅用于账单占用与验证

4) Active（活期可用）
- 未被账单占用，可用于新账单或撤销锁仓

5) Reserved（账单占用）
- 已进入账单流程，占用中，不可撤销锁仓

==================================================
4. 强制不变式（必须写入合约断言/测试）
==================================================
Invariant-1（1:1 映射）
- 对任意 user + token：
  mappedTotal == lockedReal

Invariant-2（状态守恒）
- active + reserved == lockedReal

Invariant-3（账单占用约束）
- 任意处于 Pending/Confirmed/Disputed 状态账单的 amount 必计入 reserved

Invariant-4（不可越权划转）
- 无有效 billId 与状态的 transfer intent 必须 revert

Invariant-5（终态不可回退）
- Settled/Cancelled/Expired/Resolved* 状态不可回退

==================================================
5. 账单状态机（标准）
==================================================
状态：
- Pending
- Confirmed
- Disputed
- Settled
- Cancelled
- Expired
- ResolvedBuyer
- ResolvedSeller
- SplitResolved

允许迁移：
- Pending -> Confirmed
- Pending -> Cancelled
- Pending -> Expired
- Confirmed -> Settled
- Confirmed -> Disputed
- Disputed -> ResolvedBuyer / ResolvedSeller / SplitResolved

禁止迁移：
- 任意终态 -> 非终态

==================================================
6. 资金与映射规则（核心）
==================================================
1) 用户开启锁仓模式：
- 用户确认锁仓额度 L
- 协议记录 lockedReal += L
- 映射额度 mappedTotal += L
- 初始 active += L, reserved += 0

2) 创建账单 amount=A：
- require active >= A
- active -= A
- reserved += A
- bill.status = Pending

3) 确认账单：
- bill.status: Pending -> Confirmed
- reserved 不变

4) 结算账单：
- require bill.status == Confirmed（或仲裁允许结算状态）
- require allowance/balance 条件满足
- transferFrom(payer, payee, A)
- reserved -= A
- lockedReal -= A
- mappedTotal -= A
- bill.status = Settled

5) 取消/过期账单：
- reserved -= A
- active += A
- bill.status = Cancelled/Expired

6) 撤销锁仓（用户主动）：
- 仅允许从 active 撤销
- requestedUnlock <= active
- active -= requestedUnlock
- lockedReal -= requestedUnlock
- mappedTotal -= requestedUnlock

==================================================
7. 非账单划转攻击定义与链上记录
==================================================
攻击行为定义：
- 非法调用资金划转函数（无 billId 或 bill 状态不合法）
- 伪造/重复 billId 结算
- 越权执行 confirm/cancel/settle

处理规则：
- 必须 revert
- 必须 emit 安全事件（记录请求方、目标、金额、原因）

建议事件：
- InvalidTransferIntent
- UnauthorizedBillAction
- DuplicateSettlementAttempt

==================================================
8. 权限与角色
==================================================
角色：
1) User/Payer（资金所有者）
2) Payee（收款方）
3) Agent（代表用户执行）
4) Arbitrator（仲裁角色，可选）
5) Governance/Timelock（参数治理）

权限规则：
- 创建账单：授权 agent 或 payer
- 确认/取消：按账单策略授权方
- 仲裁裁决：仅仲裁角色
- 参数修改：仅 timelock/governance

==================================================
9. 前端与交互规范（傻瓜式）
==================================================
首次上手四步：
1) 连接钱包
2) 一键开启锁仓模式（额度 + 有效期）
3) 启动 Agent 工作
4) 在账单面板确认/撤销/仲裁

必备面板字段：
- 锁仓总额 lockedReal
- 活期 active
- 占用 reserved
- 账单列表（状态、金额、时间、对手方、操作按钮）

必备规则提示：
- “资金在你钱包中”
- “仅账单流程可触发划转”
- “未占用额度可撤销锁仓”

==================================================
10. 安全要求（必须）
==================================================
1) settle 路径防重入
2) checks-effects-interactions 严格执行
3) 非标准 ERC20 兼容处理
4) 所有关键状态迁移发事件
5) 幂等保护（重复调用安全）
6) 授权最小化（优先 permit/短期授权）

11. 测试与验收标准
==================================================
A. 功能测试
1) 开启锁仓 -> 映射1:1
2) 创建账单占用 active->reserved
3) 确认后结算成功且状态正确
4) 取消/过期释放 reserved
5) 仅 active 可撤销锁仓

B. 并发测试
1) 同用户多账单并发占用一致性
2) 多对手方并发账单（A/B/C/D/E/F）状态同步
3) 重复结算与竞态下无双花

C. 安全测试
1) 重入攻击模拟
2) 越权调用拦截
3) 非账单划转拦截+日志
4) 非标准 token 行为测试

D. 验收门槛（建议）
1) 关键路径测试 100% 通过
2) 高并发压力下状态不变式无破坏
3) 失败分类准确率 >= 95%
